# Product

## Register

product

## Users

Developers and design-minded engineers evaluating or adopting Glaise — a fixed-skin design
system delivered as agent skills. They meet the project through its public README gallery
(`assets/gallery/*.webp`) before ever installing it. Fluent in the category's best tools
(Linear, Vercel, Raycast, Stripe); they judge the system by whether its own demo could ship
inside one of those products. The demo screen itself is a fictional ops/monitoring console
("Acme Ops") — the archetypal Glaise product surface.

## Product Purpose

Glaise gives every product built with it one shared, non-negotiable skin (light-first,
monochrome primary, floating panels, Inter, Lucide, headless primitives) while the product's
soul — layout, hierarchy, signature — stays free. The gallery sample is the family's public
face: proof the engine produces premium UI. Success = a category-fluent developer sees the
gallery and trusts the system immediately.

## Brand Personality

Quietly luxurious, dense, technical. Software-craft documentation, not marketing gloss.
Weightless surfaces, exact alignment, zero decoration.

## Anti-references

- AI-slop dashboards: lavender-on-dark gradients, spotlight cards, glassmorphism.
- Ghost cards (border + shadow stacked), side-stripe selected states, pill CTAs.
- Placeholder art: empty dots/squares standing in for icons, lorem-ipsum data.
- Bootstrap/Material default look — any styled UI kit aesthetic.

## Design Principles

1. The skin is law; creativity lives in composition, hierarchy, and the signature.
2. Weightless depth — separation by value, not by outline (light cards are edge-free).
3. Semantic color is earned by meaning, never decorative; the default skin is monochrome.
4. Density is a feature; whitespace is structure, not emptiness.
5. Practice what the docs preach — the demo must pass glaise-review's own bars.

## Accessibility & Inclusion

WCAG AA enforced mechanically (contrast.mjs over the effective tokens, both themes) plus the
semantic-separation rule (danger ≥30° in hue from a chromatic primary). Keyboard focus must
be visible (2px primary-focus outline). Touch targets ≥44px on touch viewports.
